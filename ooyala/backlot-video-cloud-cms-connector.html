    Connecteur CMS Backlot/Video Cloud

    L' CMS API adaptateur Backlot prend en charge un ensemble sélectionné d' Backlot API opérations afin de vous aider à maintenir le fonctionnement des intégrations et des applications pendant la transition vers les API Video Cloud. Pour en savoir plus sur les API de la plate-forme Video Cloud, consultez Mise en route avec les API Brightcove.

    Remarque : cette rubrique s'applique uniquement aux clients Ooyala migrés vers Video Cloud.

    Exigences

    Notez que pour que l'adaptateur de lecture fonctionne, vous devrez travailler avec votre CXM pour effectuer des ajustements aux paramètres DNS sur vos ordinateurs et périphériques. Pour plus de détails, reportez-vous à la section Ajout du serveur DNS Brightcove .

    Points de terminaison des ensembles d'annonces

    Video Cloud n'a pas le concept d'un ad_set, mais les données de jeu d'annonces associées à une vidéo seront migrées vers cue_points, et le ad_set_id sera stocké dans un champ personnalisé de cette pour nous permettre d'identifier les vidéos associées à un jeu d'annonces.

    Voir Migration vers Video Cloud : Ensembles d'annonces pour plus d'informations sur la façon dont les données des jeux d'annonces sont mappées aux points de cueils pour différentes sources d'annonces.

    Le tableau ci-dessous montre ce que Backlot API ad_sets les opérations sont prises en charge.

    ad_sets Points de terminaison
    Point de terminaison API de backlot Méthode Prise en charge Remarques
    v2/ad_sets GET
    • Video Cloud n'a pas le concept de jeux d'annonces ; certaines informations associées à chaque jeu d'annonces sont migrées en tant que données de point de cueil pour les vidéos associées à l'ensemble d'annonces. Voir Jeux d'annonces pour plus d'informations
    POST
    • Video Cloud n'a pas le concept de jeux d'annonces, de sorte que la tentative de création d'un jeu d'annonces échouera
    v2/ad_sets/{set_id} GET
    • Video Cloud n'a pas le concept de jeux d'annonces ; certaines informations associées à chaque jeu d'annonces sont migrées en tant que données de point de cueil pour les vidéos associées à l'ensemble d'annonces. Voir Jeux d'annonces pour plus d'informations
    PATCH
    • Cette demande mettra à jour les données de cuepoint pour les vidéos pertinentes
    DELETE
    • Cette demande supprimera les données de point de cueil pour les vidéos pertinentes.
    v2/ad_sets/{set_id}ads POST
    • Ajoute un point de cueil aux vidéos pertinentes
    DELETE
    • Supprime un point de cueille des vidéos pertinentes

    Points de terminaison des actifs

    Le tableau ci-dessous montre ce que Backlot API asset les opérations sont prises en charge.

    Notez que l' API Query qui fournit le filtrage des opérations d'actifs à l'aide d'instructions SQL n'est que partiellement prise en charge. where et les orderby requêtes ne sont pas prises en charge. limit les requêtes devraient fonctionner.

    Points de terminaison des actifs
    Point de terminaison API de backlot Méthode Prise en charge Remarques
    /v2/assets GET
    • ad les vidéos de type seront des vidéos régulières dans Video Cloud avec le tag ooyala_my_ads
    • youtube les vidéos de type ne seront pas migrées vers Video Cloud
    • external_id est en cours de migration vers Video Cloud, mais sera un champ en lecture seule disponible pour l'API de lecture (pour les lecteurs Ooyala fournissant du contenu Video Cloud)
    • Les restrictions de temps récurrentes et les restrictions IP ne sont pas migrées vers Video Cloud
    POST
    • Voir Ingestion à l'aide de l'API Backlot pour plus de détails
    • L'ingestion de Video Cloud nécessite que le chunk_size soit 5 Mo ou plus (sauf pour le dernier segment) - une erreur sera renvoyée si la valeur chunk_size est définie sur la valeur inférieure
    • Étant donné que les vidéos de youtube type ne sont pas prises en charge dans Video Cloud, la tentative de création de vidéos de ce type renvoie une erreur. ad peuvent être créés comme des vidéos régulières ayant la balise ooyala_my_ads.
    • youtube les vidéos de type ne seront pas migrées vers Video Cloud

    • youtube les vidéos de type ne seront pas migrées vers Video Cloud

    • Mise à jour uniquement des propriétés via PATCH
    PATCH
    • Seules les propriétés migrées vers Video Cloud peuvent être mises à jour voir Backlot et Video Cloud : Champs vidéo et liste de lecture
    DELETE Notez que Video Cloud n'autorise pas la suppression d'une vidéo si elle appartient à des listes de lecture
    /v2/assets/{asset_id}/ad_set GET
    • Les jeux d'annonces seront migrés vers Video Cloud en tant que points de cueils
    /v2/assets/{asset_id}/closed_captions GET
    • Cette demande renvoie une réponse HTTP 302 avec l'emplacement défini sur le contenu des sous-titres - notez que Video Cloud retournera WebVTT, le format standard pour les sous-titres en HTML5, plutôt que DFXP.
    PUT
    • Remplacez la piste de texte WebVTT existante ; notez que vous pouvez télécharger des légendes WebVTT ou DFXP - DFXP sera converti en WebVTT

      Note : Les légendes DFXP migrées vers Video Cloud sont converties en pistes de texte WebVTT, comme c'est ce que Brightcove Player utilise. Deux notes importantes sur la migration des données :

      • Si plusieurs sous-titres linguistiques sont stockés dans le DFXP, il sera divisé et transformé en plusieurs pistes de texte WebVTT pour les différentes langues. Les pistes de texte anglais seront définies par défaut. Vous pouvez modifier la langue par défaut dans Studio si nécessaire.
      • Étant donné que les pistes de texte nécessitent un identificateur de langue, si le fichier DFXP n'en contient pas, en seront affectés aux sous-titres WebVTT convertis. Si ce n'est pas la langue correcte, vous devrez la corriger en modifiant les propriétés de la piste de texte dans Studio afin que le menu sous-titrage affiche la langue correcte.
    DELETE  
    /v2/assets/{asset_id}/drm_attributes/{drm_type} GET
    • La DRM dans Video Cloud est appliquée à toutes les vidéos d'un compte DRM, à moins qu'elles ne soient signalées pour ne pas être emballées avec un drm_disabled champ
    PATCH
    • La DRM dans Video Cloud est appliquée à toutes les vidéos d'un compte DRM, à moins qu'elles ne soient signalées pour ne pas être emballées avec un drm_disabled champ
    /v2/assets/{asset_id}/generated_preview_images GET
    • Video Cloud ne stocke pas plusieurs images miniatures
    /v2/assets/{asset_id}/labels GET
    • Tous labels seront migrés vers tags , donc cette demande retournera les balises vidéo
    POST
    • Tous labels seront migrés vers tags , donc cette requête ajoutera une balise vidéo. Notez que la mise à jour des étiquettes directement dans Video Cloud n'est pas prise en charge.
    /v2/assets/{asset_id}/labels/{label_id} PUT
    • Cela remplacera l'étiquette (a tag) dans Video Cloud. Notez que la mise à jour des étiquettes directement dans Video Cloud n'est pas prise en charge.
    DELETE
    • Cela supprimera l'étiquette (a tag) dans Video Cloud. Notez que la mise à jour des étiquettes directement dans Video Cloud n'est pas prise en charge.
    /v2/assets/{asset_id}/metadata GET
    • metadata seront migrés vers tags dans le formulaire meta:;:{key}:;:{value}. Notez que la longueur totale de chaque balise est limitée à 128 caractères sur un octet.
    PUT
    • metadata seront migrés vers tags dans le formulaire meta:;:{key}:;:{value} - cette demande ajoutera ou remplacera les balises dans ce format. Notez que la longueur totale de chaque balise est limitée à 128 caractères sur un octet.
    PATCH
    • metadata les éléments seront migrés tags dans le formulaire meta:;:{key}:;:{value} - cette demande modifiera les balises dans ce format
    DELETE
    • metadata éléments seront migrés vers tags dans le formulaire meta:;:{key}:;:{value} - cette demande supprimera toutes les balises dans ce format
    /v2/assets/{asset_id}/movie_urls GET
    • movie_urls sont sources dans Video Cloud
    POST
    • Fonctionne de la même manière dans Video Cloud - cette demande va ajouter une télécommande rendition (et donc source ) à la vidéo
    /v2/assets/{asset_id}/movie_urls/{type} DELETE
    • Cette demande supprime uniquement les formats associés distants du type spécifié
    /v2/assets/{asset_id}/player GET
    • Cette requête retournera la construction du lecteur Ooyala par défaut, car Video Cloud n'associe pas les lecteurs aux vidéos ; au lieu de cela, n'importe quelle vidéo peut être associée à n'importe quelle instance de lecteur par sa vidéo id. Vous pouvez obtenir des lecteurs Video Cloud via l' API Player Management
    /v2/assets/{asset_id}/player/{player_id} GET
    • Video Cloud n'associe pas les lecteurs aux vidéos ; au contraire, n'importe quelle vidéo peut être associée à n'importe quelle instance de lecteur par sa vidéo id. Vous pouvez obtenir des lecteurs Video Cloud via l' API Player Management
    /v2/assets/{asset_id}/preview_image_urls GET
    • Video Cloud ne stocke pas plusieurs images miniatures, donc cette demande retournera l' poster image
    POST

    PUT
    • Video Cloud ne stocke pas plusieurs images miniatures, donc cette demande définira l' poster image à l'aide de la première image de la liste
    DELETE
    • Video Cloud ne stocke qu'une seule image d'affiche, et vous pouvez la remplacer, mais pas la supprimer
    /v2/assets/{asset_id}/preview_image_files/{image_id} DELETE
    • Video Cloud ne stocke qu'une seule image d'affiche, et vous pouvez la remplacer, mais pas la supprimer
    /v2/assets/{asset_id}/preview_image_files GET
    • Video Cloud ne stocke pas plusieurs images miniatures, donc cette demande retournera l'URL de l' poster image
    POST
    • Video Cloud ne stocke pas plusieurs images miniatures, donc cette demande définira l' poster image à l'aide de la première URL de l'image dans la liste comme ressource distante
    PUT
    • Video Cloud ne stocke pas plusieurs images miniatures, donc cette demande définira l' poster image en utilisant la première image de l'URL de liste dans la liste comme ressource distante
    DELETE
    • Video Cloud ne stocke qu'une seule image d'affiche, et vous pouvez la remplacer, mais pas la supprimer
    /v2/assets/{asset_id}/primary_preview_image GET
    • Video Cloud ne stocke qu'une seule image d'affiche, donc cette demande retournera

    • Video Cloud ne stocke qu'une seule image d'affiche, donc cette demande remplacera l'affiche

    /v2/assets/{asset_id}/closed_captions GET
    • Retourne la valeur par défaut text_track de la vidéo. Les vidéos migrées n'en auront qu'une text_track - d'autres peuvent être ajoutées via Video Cloud. Si vous ajoutez d'autres pistes de texte, définissez celle par défaut pour qu'elle soit renvoyée pour cette demande.

      Note : Les légendes DFXP migrées vers Video Cloud sont converties en pistes de texte WebVTT, comme c'est ce que Brightcove Player utilise. Deux notes importantes sur la migration des données :

      • Si plusieurs sous-titres linguistiques sont stockés dans le DFXP, seules les légendes anglaises seront stockées dans les pistes de texte WebVTT, car WebVTT ne prend pas en charge plusieurs sous-titres linguistiques dans un seul fichier.     Étiquette les terminaux

    Les étiquettes seront migrées vers Video Cloud en tant que tags. Contrairement aux étiquettes, les balises sont définies sur des vidéos individuelles - il n'y a pas de « jeu de balises » au niveau du compte, et les balises ne sont pas hiérarchiques. Les hiérarchies d'étiquettes seront préservées dans les balises Video Cloud comme ceci :

    En dehors de ces différences, les balises peuvent être utilisées de nombreuses façons que les étiquettes sont, par exemple :

    • recherche de vidéos qui ont une partie, la totalité ou aucune d'un ensemble de balises
    • Sélection de vidéos pour une playlist « intelligente » à l'exécution
    Notez que si tous les /labels terminaux sont pris en charge, certaines opérations, en particulier pour les comptes qui ont un grand nombre de vidéos, prendront plus de temps.

    Le tableau ci-dessous montre ce que Backlot API labels les points de terminaison sont pris en charge.

    Point de terminaison API de backlot Méthode Prise en charge Remarques
    /v2/labels GET Les étiquettes apparaîtront comme tags dans Video Cloud
    /v2/labels POST Les étiquettes apparaîtront comme tags dans Video Cloud
    /v2/labels/by_full_path/{paths} GET Les étiquettes apparaîtront comme tags dans Video Cloud
    /v2/labels/by_full_path/{paths} POST Les étiquettes apparaîtront comme tags dans Video Cloud
    /v2/labels/{label_id} GET Les étiquettes apparaîtront comme tags dans Video Cloud
    /v2/labels/{label_id} PATCH Les étiquettes apparaîtront comme tags dans Video Cloud
    /v2/labels/{label_id} DELETE Les étiquettes apparaîtront comme tags dans Video Cloud. Notez qu'une étiquette ne sera pas supprimée si elle est référencée par une liste de lecture.
    /v2/labels/{label_id}/children GET Les étiquettes apparaîtront comme tags dans Video Cloud
    /v2/labels/{label_id}/assets GET Les étiquettes apparaîtront comme tags dans Video Cloud

    Points de terminaison des règles de publication

    Les restrictions de lecture mises en œuvre par les règles de publication sont appliquées aux ressources individuelles dans Video Cloud. Les restrictions de domaine ne sont pas migrées, car dans Video Cloud, elles sont définies sur les lecteurs et non sur les vidéos.

    Sur les ressources vidéo
    • géo-restrictions
    • planification (date de début et de fin)

    Les propriétés des règles de publication Ooyala relatives aux géo-restrictions et à la planification seront appliquées aux vidéos associées à la règle de publication et id seront stockées dans un champ personnalisé portant le nom publishing_rule_id. Les opérations de règle de Backlot API publication fonctionnent sur toutes les vidéos associées à la règle de publication, mais seules la planification ou les restrictions géographiques seront renvoyées.
