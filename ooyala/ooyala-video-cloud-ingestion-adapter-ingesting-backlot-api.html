    Ooyala-Video Cloud Ingestion Adaptateur : Ingestion avec l'API Backlot

    Cette rubrique traite de l'utilisation de l' Backlot API pour ingérer du contenu dans Video Cloud.

    Remarque : cette rubrique s'applique uniquement aux clients Ooyala migrés vers Video Cloud.

    Introduction

    Utilisez le Backlot API pour télécharger du contenu par programme dans Video Cloud. Cette approche vous permet d'intégrer votre système de gestion de contenu (CMS) ou vos flux de travail directement à la plateforme Video Cloud.

    Notez que vous pouvez également migrer vos intégrations ou applications vers l'API CMS et l'API Dynamic Ingest à tout moment.

    Vous pouvez utiliser le Backlot API pour télécharger et gérer par programme des vidéos, des miniatures et des fichiers de sous-titres codés.

    Si vous souhaitez ingérer du contenu à l'aide de l'API Backlot, consultez :

    • Ressources vidéo et audio (pour les vidéos et les miniatures)
    • Utilisation des sous-titres codés

    L'appel principal utilisé pour télécharger des fichiers (appelés « assets ») est l' /v2/assets itinéraire.

    Note : N'ajoutez pas de vidéo avec un null external_id, c'est-à-dire un external_id sans valeur ("") ou une valeur de "null". De tels ID externes null ne peuvent pas être recherchés ultérieurement.

    Ressources vidéo et audio

    Lorsque vous téléchargez une vidéo sur Ooyala, Ooyala la transcode automatiquement en utilisant les paramètres spécifiés pour votre compte et rend les vidéos disponibles pour la lecture via votre lecteur.

    Notez que les ressources audio seront stockées dans Video Cloud sous forme de vidéos sans piste vidéo.

    Nous vous recommandons de télécharger des vidéos sur Ooyala dans H.264, mais nous prenons également en charge de nombreux codecs couramment utilisés.

    Pour créer de nouvelles ressources vidéo ou audio :
    1. Créer l'actif avec des propriétés de base
    2. Récupérer les « URL de téléchargement » du système
    3. Pour chaque « URL de téléchargement », mettez le morceau de fichier approprié sur le système
      Note : Lorsque vous téléchargez chaque morceau, ne signez pas la demande, comme vous le feriez d'autres requêtes. Il suffit de mettre le contenu du bloc aux URL exactes. Ces URL ont déjà été signées pour vous.

    Obtenir le téléchargement des URL pour les ressources nouvellement créées

        [GET]/v2/assets/{asset_id}/uploading_urls
    Note : Lorsque vous téléchargez chaque morceau, ne signez pas la demande, comme vous le feriez d'autres requêtes. Il suffit de mettre le contenu du bloc aux URL exactes. Ces URL ont déjà été signées pour vous.

    Définir le statut de chargement

        [PUT] /v2/assets/ {asset_id} /upload_status
    {
       "status":"uploaded"
    }

    Afficher les flux d'une vidéo

    Le terme flux dans ce contexte désigne les processus Internal-to-Ooyala qui sont démarrés par le système pour transcoder vos vidéos.

    Note : Vous ne pouvez pas contrôler ou manipuler ces flux ; ils sont contrôlés par le système. Vous ne pouvez voir que leurs noms, pas les changer.
        [GET] /v2/assets/ {asset_id} /stream

    Afficher les fichiers source d'une vidéo

        [GET]/v2/assets/{asset_id}/source_file_info

    Obtenir les URL pour les images d'aperçu générées automatiquement

        [GET] /v2/assets/ {asset_id} /generated_preview_images

    Télécharger une image d'aperçu personnalisée

        [POST] /v2/assets/ {asset_id} /preview_image_files
    <fichier_contenu >

    Définir la configuration d'image d'aperçu principal

    Définissez le type de l'image d'aperçu principale d'une ressource sur l'une des options suivantes :
    • generated: utiliser l'image d'aperçu générée automatiquement
    • uploaded_file: utilisez l'image d'aperçu personnalisé téléchargée
    • remote_url: URL de l'image d'aperçu
        [PUT] /v2/assets/ {asset_id} /primary_preview_image
    {
       « type » :"généré » | « uploaded_file » | « remote_url »
    }

    A propos des mét

    Un actif peut avoir jusqu'à 100 paires nom/valeur de métadonnées personnalisées. Un nom est limité à 128 caractères et sa valeur est limitée à 2 048 caractères.

    Exemples

    Cet exemple crée une vidéo : 
        [POST] /v2/actifs {
       « name » :"Blaireau de miel »,
       « nom_fichier » : » honeybadger.avi « ,
       "asset_type":"video",
       « file_size » :398422,
       « chunk_size » :100000
    }

    Video Cloud renvoie une réponse similaire à la suivante :

       « preview_image_url » :null,

       « description » :null,
       « status » :"téléchargement »

        [PUT] /v2/assets/ {asset_id} /upload_status

    Utilisation des sous-titres codés

    Vous pouvez ajouter des sous-titres codés à des vidéos ou à des vidéos de ressources distantes via l'API Backlot.

    Formats supportés pour les sous-titres

    formats pris en charge
    Extension de fichier Description
    .vtt (WebVTT/) - this is the format required for HTML5 video players WebVTT peut être utilisé pour les sous-titres embarqués et sidecar dans des formats de streaming HTTP tels que HLS et DASH.
    .TTML ( anciennement DXFP ) - sera converti en WebVTT TTML (Fichier de langage de balisage de texte chronométré. Généralement, un fichier de sous-titres de ce type aura une extension .dfxp (Distribution Format Exchange Profile).

    Charger les sous-titres

        [PUT] /v2/assets/ {asset_id} /closed_captions/ {dfxp_file_content}

    Obtenir des sous-titres

        [GET] /v2/assets/ {asset_id} /sous-titres

    Mettre à jour les sous-titres

        [PUT] /v2/assets/ {asset_id} /sous-titres

    Supprimer les sous-titres

        [DELETE] /v2/assets/ {asset_id} /closed_légendes
